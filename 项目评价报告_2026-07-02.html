1. 项目概览与业务定位

业务背景：长城汽车·长城保理公司精工板块每日需将 5 个数据源的 16 个有色金属/原油品种价格填入 Excel 共享表。原流程纯人工 + Excel 操作员手工填表，每日耗时 1-2 小时。

项目目标：自动抓取 + 一次性写入 Excel，将人工介入压到 10 分钟以内（仅保持 SMM / 亚洲金属网登录态）。

3. 架构分析

3.1 双轨架构（最大问题）

项目存在两套并行实现，互不复用：

轨道 A：fill_and_verify.py（实际在用）
  • 单文件 780 行，含数据采集 + 填表 + 历史核查 + 截图 + OCR 备选
  • 调用 jinggong_monitor/fetcher_ccmn.py、fetcher_asianmetal.py、fetcher_tungsten.py（复用了 3 个）
  • SMM 抓取在本文件内重写（get_smm_cdp 正则提取，行 145-182）
  • WTI 抓取在本文件内重写（get_wti_estimate，行 322-337）
  • 填表逻辑在本文件内（fill_sheet1 / fill_sheet2）
  • 价格校验在本文件内（historical_avg + 50% 阈值）
轨道 B：main.py + orchestrator.py + excel_filler.py（设计良好但未用）
  • 模块化设计：BaseFetcher 抽象 + PriceCollector 调度 + 代理品种映射 + 多源择优
  • 支持 22 品种（含钢板/不锈钢/铁合金等，超出当前 16 品种需求）
  • 有 PriceValidator 多源交叉验证、SpreadCalibrator 期现价差标定
  • 致命问题：excel_filler._EXCEL_PATH 硬编码到 ~/Desktop/工作/招聘/...xlsx，根本跑不通
  • run.sh 调用 main.py，但 README/SKILL 都说实际用 fill_and_verify.py — 入口不一致

5. 关键问题清单（按严重程度）

P0 · 双轨架构 + 入口不一致
run.sh 调 main.py（轨道 B），但 README/SKILL 都说实际跑 fill_and_verify.py（轨道 A）。新接手的人跑 bash run.sh 会触发轨道 B，而轨道 B 的 excel_filler._EXCEL_PATH 指向错误路径，直接跑不通。
影响：新人接手即踩坑；维护两套代码浪费精力。
P0 · cron 未配置，自动化未落地
README 明确写「三个 cron（15:00 WTI / 17:00 主流程 / 21:00 钨粉补查）」，但 crontab -l 查无任何 jinggong 相关配置。当前实际是「主人手动触发」。
影响：项目核心价值（自动化）未实现；7/1 行大面积缺失（11/17 空）就是证据。
P1 · 硬编码绝对路径遍布
/Users/siqi/Desktop/AI/jinggong-commodity-monitor/... 散落在 fill_and_verify.py（行 33, 39, 254）、fetcher_smm.py（行 47）、fetcher_asianmetal.py（行 34）等处。换机器、换用户、换目录全部跑不通。
建议：统一用 Path(__file__).parent 或环境变量 JINGGONG_PROJECT_ROOT。
P1 · 无单元测试
项目根无 tests/ 目录，无 pytest 配置。_parse_price_range 这种核心解析函数没有任何自动化测试覆盖。本次评价手工跑的 11 个 case 应该固化成测试。
影响：任何改动都可能悄无声息地破坏解析逻辑。
P2 · SMM 双方案并存
fill_and_verify.py 内 get_smm_cdp（正则提取，行 145-182）与 jinggong_monitor/fetcher_smm.py（Cookie + Playwright DOM 提取）是两套独立实现。实际只用前者，后者成为僵尸代码。
建议：删 fetcher_smm.py 或将其逻辑合并到 fill_and_verify。
P2 · OCR 备选用免费 key
fill_and_verify.py:269 的 OCR.space 用了示例 key "helloworld"，免费配额极低（每天 25 次左右）且不稳定。亚洲金属网价格表改图片渲染后，OCR 是兜底方案，但这个 key 不可靠。
建议：注册正式 API key 或换本地 OCR（PaddleOCR）。
P2 · Excel junk row 未清理
Row 120 全空行留在表里，fill_and_verify.py 的 manual 模式只清理 116-118 行（行 406-408），没覆盖 120。
影响：历史核查、图表都可能受空行干扰。
P3 · 空目录残留
fetchers/、monitor/、output/、pipeline/、工作/ 都是空目录，是早期目录结构遗留。
建议：删除或归档。
P3 · 旧 Excel 备份文件散落
2026年有色金属市场价格共享.before-test-1623.xlsx、(2).before-redo-1713.xlsx、(2).xlsx.bak.20260626_162221 三个备份散在根目录。
建议：移到 backup/ 子目录或定期清理。

6. 亮点

文档质量出色（9.5/10）
README + SKILL.md 合计 ~35KB，覆盖业务背景、品种映射、数据源详解、执行流程、故障处置手册、入口密码本、接口契约表、踩坑教训。
特别赞：SKILL.md 附录 B「故障处置手册」按症状查处置，附录 A「接口契约表」让别的 AI 拿 SKILL 接活不用问主人。这种文档完整度在内部项目里非常少见。
踩坑沉淀充分
金属硅中间价字段映射规则（H/I/L 列分别取 avgPrice/minPrice/maxPrice，而不是字面牌号）、officecli watch session 互覆盖问题、中钨在线 HTTP vs HTTPS、ccmn AJAX 端点发现 — 这些都是非常硬核的实战经验，全部沉淀到文档，避免重踩。
价格校验机制设计合理
fill_and_verify.py 的 historical_avg + 50% 阈值校验，6/29 实战拦截了 ccmn 返回 MN=6 的采集错误（实际 19200）。这种「写入前与历史对比」的防御性设计很到位。
现场截图与证据保存
每个数据源抓取后自动截图/存 HTML 到 screenshots/{日期}/，失败也截一张标 ❌ 的图。这种「留证据便于追溯」的设计在数据采集项目里是最佳实践。
核心公开源稳定可用
ccmn / akshare / 中钨在线 三个公开源实测全部通过，ccmn AJAX 端点设计优雅（一次 POST 拿全 36 品种），价格解析函数 11/11 测试通过。
